# CLAUDE.md

## 项目结构

- `cXX/` — 各章节的实验配置、脚本与答案
- `elasticsearch-course/` — 课程 Markdown 正文
- `.markdownlint-cli2.yaml` — 仓库级 Markdown 检查配置

## 提交规范

### 语言

提交消息必须使用英文。

### 格式

```text
<type>(<scope>): <short summary>

- <change point 1>
- <change point 2>
```

正文使用 `-` 开头的 bullet list。

### 类型（type）

优先使用：`feat`、`fix`、`refactor`、`perf`、`test`、`docs`、`build`、`ci`、`chore`、`revert`。
无合适匹配时默认 `chore`，不自行发明新类型。

### 范围（scope）

优先按模块选择 scope（如 `data-ingestion`、`container-management`），跨多个模块且无统一主题时使用 `multi`。使用小写 kebab-case。

### 正文

- 每条 bullet 描述一个独立变更点，不重复
- 仅描述实际代码/内容变更，不包含对话反馈、review 意见等过程性描述

### 反引号处理

提交消息含反引号时，使用单引号 `-m '...'` 或 `git commit -F` 传递，禁止使用双引号 `-m "..."`（避免 shell 命令替换）。

## 提交前检查

### 1. 敏感信息

检查暂存区 diff 中是否包含密码、私钥、access token、API key、凭据、个人信息、公网 IP、环境变量文件等。发现敏感信息时停止提交，不纳入版本控制。

### 2. 格式化

对提交范围内的文件进行格式化：

- 优先使用仓库内已定义的格式化规则、命令、脚本、hook 或配置文件。
- 如果仓库未定义，使用该语言或文件格式的官方格式化工具或风格。
- 如果不存在官方标准，遵循广泛接受的行业惯例。
- 格式化范围限定在提交范围内的文件，不引入无关的全仓库变更。
- 格式化工具不可用、配置冲突、或格式化可能带来风险时，停止并告知用户，不跳过检查、不自行安装工具。

### 3. Lint

对提交范围内的文件进行检查：

- 优先使用仓库内已定义的 lint 规则、命令、脚本、hook 或配置文件。
- 如果仓库未定义，使用该语言或文件格式的官方 linter。
- 如果不存在官方 linter，使用广泛接受的 linter 或行业惯例。
- 先格式化后 lint（除非仓库明确定义了不同顺序）。格式化与 lint 是独立要求，一项通过不代表另一项可跳过。
- lint 范围和自动修复限定在提交范围内的文件，不引入无关的全仓库变更。
- Linter 不可用、配置冲突、需要安装工具、或修复可能带来风险时，停止并告知用户，不跳过检查、不削弱规则、不自行安装工具。
- 提交前报告实际执行的 lint 命令及结果。如果经检查仓库配置和常规工具后仍不存在可用 linter，明确报告无可用的 lint 命令，并停止等待用户指示。

### 4. 临时文件

- 临时文件统一放在 `.agent-tmp/`
- 任务完成后清理已跟踪的临时文件
- 临时文件不得纳入提交
