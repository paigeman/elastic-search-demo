# 00-study-guide 练习与验收答案

## 第 1 题

**题目：** 写出一个适合 Elasticsearch 的业务场景和一个不适合的场景。

### 适合的业务场景：电商商品搜索

电商平台需要让用户根据商品名称、描述和品牌进行全文搜索，同时按照分类、价格区间、库存状态等条件筛选商品，并按相关性或销量排序。这个场景适合使用 Elasticsearch，因为它擅长全文检索、多条件过滤、相关性排序和数据聚合，能够为大量商品提供快速、近实时的搜索体验。

### 不适合的业务场景：银行账户转账

银行转账需要保证扣款和入账要么同时成功，要么同时失败，还需要严格的一致性、事务和约束能力。Elasticsearch 不适合作为这类业务的唯一数据源，因为它不是以复杂事务、强一致性和关系约束为核心设计的。此场景更适合使用支持 ACID 事务的关系型数据库；如果业务需要搜索转账记录，可以再将数据同步到 Elasticsearch 中用于检索和分析。

## 第 2 题

**题目：** 解释 `9200`、`9300`、`5601` 三个端口的用途。

- `9200`：Elasticsearch 的 HTTP API 端口。应用程序、Kibana 和 `curl` 等客户端通过该端口访问 Elasticsearch，执行查询、写入和管理等操作。
- `9300`：Elasticsearch 集群中各 ES 节点之间的传输（transport）通信端口，不用于 Elasticsearch 与 Kibana 之间的通信。单节点部署没有跨节点通信需求，通常无需对外映射或开放该端口；多节点部署时也只应在集群内部开放。
- `5601`：Kibana 的 Web 服务端口。用户通过浏览器访问该端口使用 Kibana，Kibana 再通过 `9200` 访问 Elasticsearch。

它们之间的通信关系可以简化为：

```text
浏览器 ──> Kibana:5601 ────────────> Elasticsearch:9200

curl ──────────────────────────────> Elasticsearch:9200
后端应用 ──────────────────────────> Elasticsearch:9200
浏览器 ──> 业务后端 API ──────────> Elasticsearch:9200

Elasticsearch 节点 A:9300 <──> Elasticsearch 节点 B:9300
```

`curl`、Kibana 和后端应用直接访问 Elasticsearch 的 `9200` 是常见用法。普通业务前端通常不直接访问 `9200`，而是通过业务后端间接访问，以避免暴露 Elasticsearch 地址和凭据，并由后端统一进行权限控制、参数校验和查询限制。
