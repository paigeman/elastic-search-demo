# 20｜结课项目与验收

## 项目：企业商品搜索平台

从空环境交付一个可演示、可运维的商品搜索服务。不要只提交“能查到数据”的截图，需覆盖部署、数据、查询、安全、监控和恢复。

## 1. 业务要求

- 至少 10 万条模拟商品，字段包含商品标识、名称、描述、品牌、类目、价格、库存、标签、上下架状态、创建时间和动态属性。
- 支持关键词检索，名称权重大于描述。
- 支持品牌/类目/价格/有货过滤，价格和时间排序。
- 支持稳定翻页、高亮和品牌/价格区间聚合。
- 写入链路能够规范类目、转换价格、记录摄取时间并隔离坏数据。
- 映射更新可以通过新索引、重新索引和别名切换上线。

## 2. 基础设施要求

开发环境可以使用 Docker 或 Podman Compose；验收时需说明生产化还需要哪些变化。至少提交：

- 版本明确的 Elasticsearch 与 Kibana 配置。
- 持久卷、资源限制、健康检查和安全连接。
- 索引模板、组件模板、摄取管道和读写别名。
- 不使用 `elastic` 超级用户的读 API 密钥与写 API 密钥。
- 快照仓库和一次恢复到新索引名的证据。

## 3. 数据与查询验收

准备至少 20 条代表性查询，包括：

- 精确商品标识、品牌和类目过滤。
- 中文关键词、英文型号、拼写差异、无结果查询。
- 多条件组合、价格边界、缺失字段。
- 第 1 页和使用 PIT 与 `search_after` 的后续页面。
- 高基数聚合的受限方案。

为其中 10 条查询标注期望的前 5 条结果，记录调优前后的结果。至少解释一次分析器或映射改动，以及一次字段权重调整的依据。

## 4. 故障演练

完成并记录：

1. 单节点副本导致黄色状态，使用分配解释接口正确说明原因。
2. 提交错误价格，通过管道模拟或死信队列路径发现问题。
3. 使用只读密钥尝试写入，验证返回 403。
4. 制造一次应用超时或受控 429，验证有限退避。
5. 从快照恢复索引，验证文档数和关键查询。

不得为了演练损坏真实环境或删除未授权数据。

## 5. 运维交付物

- 架构图：客户端、入口、节点/容器、数据流、快照仓库。
- 数据字典：字段、类型、是否可搜/过滤/排序/聚合、敏感级别。
- 服务等级目标：搜索第 95/99 百分位延迟、写入延迟、可用性、恢复点目标和恢复时间目标。
- 仪表盘与告警：集群健康、磁盘、堆内存与垃圾回收、处理器、拒绝次数、延迟和快照。
- 操作手册：红色状态、磁盘告警、慢查询、凭据轮换和索引迁移。
- 容量估算：假设、实测膨胀系数、副本、余量和扩容触发点。

## 6. 评分表

| 项目 | 分值 | 达标标准 |
| --- | ---: | --- |
| 部署与可重复性 | 15 | 新机器按文档可启动，版本和依赖明确 |
| 映射与数据质量 | 15 | 类型合理、严格控制动态字段、坏数据可追踪 |
| 搜索与相关性 | 20 | 需求完整、查询集有回归、结果可解释 |
| 数据接入 | 10 | 批量操作、幂等、重试、背压和死信队列设计清晰 |
| 安全 | 10 | TLS 校验、最小权限、凭据不落库 |
| 运维与可观测性 | 15 | 告警、排障和变更流程可执行 |
| 备份与恢复 | 10 | 快照成功且实际恢复验证 |
| 性能与容量 | 5 | 有基线、服务等级目标、容量模型和余量 |

80 分以上视为能够独立应对常规企业任务；低于 80 分按失分项回到相应章节补做，而不是只背 API。

## 7. 最终自测

- 我能解释当前每个索引的负责人、用途、保留期和回滚方式吗？
- 我能从 401/403/409/429 快速缩小问题范围吗？
- 我能在不删除旧索引的情况下完成映射迁移吗？
- 我能证明备份真的可以恢复吗？
- 我能根据趋势提前扩容，而不是等磁盘达到洪泛水位吗？
