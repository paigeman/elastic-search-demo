# 01｜核心概念与架构

## 本节目标

- 理解文档、索引、节点、集群、分片和副本。
- 理解写入与搜索的基本路径。
- 避免把 Elasticsearch 当成“拥有数据表的 MySQL”。

## 1. 概念映射

| Elasticsearch 概念 | 中文含义 | 容易误解的点 |
| --- | --- | --- |
| 文档（document） | 一个 JSON 业务对象 | 字段可嵌套，但不是任意无模式 |
| 索引（index） | 同类文档的逻辑集合 | 不等于数据库；底层由多个分片构成 |
| 映射（mapping） | 字段类型与索引规则 | 类似结构定义（schema），但很多类型不能原地修改 |
| 分片（shard） | 一个 Lucene 索引，分布与扩展单位 | 分片不是越多越好 |
| 副本（replica） | 主分片的复制品 | 提升容灾和部分读取吞吐，不增加写吞吐 |
| 节点（node） | 一个 Elasticsearch 进程 | 同一机器多节点不等于真正高可用 |
| 集群（cluster） | 共享同一集群状态的节点集合 | 由具备主节点资格的节点管理元数据 |

## 2. 写入发生了什么

客户端向任意可接收 HTTP 请求的 Elasticsearch 节点发送写入；接收请求的节点在本次请求中充当协调节点，并根据文档 `_id` 将请求路由到主分片。主分片校验映射、写入并转发到副本，满足等待条件后响应。数据先进入内存结构和事务日志（translog），随后刷新（refresh）产生可搜索的段（segment），所以它是“近实时”而非每次写入后立即可搜。

协调节点不是默认固定在某一台机器上的独立角色：每个 Elasticsearch 节点都具备请求协调能力，哪个节点收到请求，哪个节点就负责这次请求的路由和结果汇总。将 `node.roles` 配置为空列表可以建立专用协调节点，使其不承担主节点、数据或摄取等职责，但小型集群通常不需要专门设置。

关键概念：

- 刷新（`refresh`）：让新数据可搜索，默认通常约 1 秒；频繁手工刷新会降低吞吐。
- 刷写（`flush`）：提交 Lucene 提交点并滚动事务日志，不等于刷新。
- 段合并（segment merge）：在后台合并不可变段，可能产生明显的磁盘输入输出负载。

## 3. 搜索发生了什么

一次搜索分为查询（query）和取回（fetch）两个阶段：协调节点把查询广播到相关分片，各分片返回候选文档及排序值，协调节点全局归并，再向命中分片取回 `_source`。`_source` 是索引文档时保存的原始 JSON 内容，本身不参与检索；倒排索引负责找到文档，`_source` 负责返回完整业务数据。因此深分页、高基数字段排序和查询过多分片都会放大处理器、内存和网络开销。

协调节点与外部负载均衡器不是同一个概念。负载均衡器在多个 Elasticsearch 入口之间分配客户端连接并处理故障切换；协调节点了解集群状态和分片位置，负责集群内部的分片路由、搜索结果归并和批量写入分发。专用协调节点可以作为负载均衡器的后端入口，但只部署一个会形成单点故障。

## 4. 一致性与高可用

- 主分片和副本不会放在同一节点。
- 单节点集群配置一个副本时会处于黄色（`yellow`）状态：主分片正常，但副本无处分配。
- 绿色（`green`）表示主、副分片均已分配；黄色（`yellow`）表示主分片正常但至少一个副本未分配；红色（`red`）表示至少一个主分片不可用。
- 高可用至少需要跨故障域的多个节点，并正确配置节点发现（discovery）、角色和副本。

查看拓扑：

```bash
escurl "$ES_URL/_cluster/health?pretty"
escurl "$ES_URL/_cat/nodes?v"
escurl "$ES_URL/_cat/indices?v&s=store.size:desc"
escurl "$ES_URL/_cat/shards?v"
```

`_cat`（compact and aligned text）是一组以紧凑表格展示集群状态的接口：`nodes`、`indices` 和 `shards` 分别查看节点、索引和分片。它适合人工查看；自动化脚本优先使用对应的 JSON API，并显式选择字段。
